# TRT Conversion Handover — Paraformer + Matcha pure TensorRT

Date: 2026-05-16 | Author: Claude (suharvest supervision)

## Overview

Goal: remove ONNX Runtime CUDA dependency from `zh_en` profile by converting
Matcha TTS and Paraformer ASR models to TensorRT engines where TensorRT can
own the high-compute path.

**Result**: Paraformer encoder+decoder fully TRT. Matcha full acoustic TRT is
still blocked by TensorRT dynamic shape limitations, but the model is now split:
duration/encoder stays in a small ORT-CPU graph and the three high-compute ODE
estimator steps run as TensorRT BF16 engines. Vocos remains TensorRT.

## Key Technical Problems & Solutions

### Problem 1: Paraformer Decoder Cask Convolution Failure

**Symptom**: `trtexec` failed at `/decoder/decoders.0/self_attn/fsmn_block/Conv` with
"Assertion isOpConsistent failed. Cask convolution isConsistent check failed."

**Root cause**: Two `make_pad_mask` subgraphs (22 nodes: Range/ReduceMax/Less/Cast)
created dynamic shape expressions that TRT couldn't resolve for the FSMN Conv layer.
The shape expression involved `BROADCAST_SIZE token_length` derived from
`ReduceMax(acoustic_embeds_len)` — a runtime dynamic value.

**Solution**: `scripts/surgery_paraformer_decoder.py` surgically removes the 22 nodes
and replaces them with external `pad_mask` [1, token_length] and `enc_pad_mask`
[1, enc_length] float32 inputs. Since CIF produces exact token counts (no padding),
masks are always all-ones.

Parity validated: ORT on modified ONNX produces identical logits and sample_ids
as the original ONNX.

### Problem 2: FP16 Decoder Produces Identical Tokens

**Symptom**: All decoder output tokens are identical (always "的" = token 1373),
regardless of input embeddings.

**Root cause**: FSMN depthwise Conv1d in FP16 loses precision. The causal convolution
accumulates over kernel_size=11, and FP16's reduced mantissa causes information loss
that makes all token positions indistinguishable.

**Solution**: Rebuild decoder with `--bf16` instead of `--fp16`. BF16 has the same
exponent range as FP32 and preserves position differentiation.

**Key insight**: Always use BF16 for FSMN-based models.

### Problem 3: Encoder BF16 TRT Produces NaN

**Symptom**: The old encoder BF16 engine produced NaN on real audio and could
fall back to ORT.

**Root cause**: The BF16 encoder plan was invalid for the deployed model on
Orin Nano; direct tests at 40/80/120/200 frames produced NaN. A separate runtime
bug forced the wrong TensorRT optimization profile for short inputs.

**Solution**: Build the encoder in FP32 with `min=40,opt=80,max=400`, keep the
decoder in BF16, and select TensorRT profiles at runtime using
`get_tensor_profile_shape("speech", profile_idx)`. The service should log
`Encoder TRT engine validated (no NaN)` and health should report
`asr_providers={"encoder":"trt","decoder":"trt"}`.

### Problem 4: Matcha Full Acoustic TRT — Shape-Tensor Detection

This is the major remaining issue. TRT 10.3's ONNX parser auto-detects shape tensors
by tracing from shape-consuming ops (Shape, Reshape target, Slice end) backward
through the graph. If a float input flows into any such chain, TRT rejects it with
"must have type Int32 or Int64."

**What was tried** (all documented in `scripts/surgery_matcha_acoustic.py`):

1. **RandomNormalLike externalization** ✅ — replaced with max-size noise [1,80,800]
   + internal Shape/Cast/Slice. ORT validated.

2. **length_scale float→int64 fixed-point** ✅ — represents `length_scale × 1000`
   as INT64, inserts Cast(FLOAT) + Div(1000.0) for float consumers. ORT validated.

3. **Range nodes replacement** ✅ — 3× Range(0, N, 1) → static constant [0..MAX] +
   Slice. ORT validated.

4. **Dynamic noise dim** — noise input changed to [1, 80, "L_mel"] instead of
   static [1, 80, 800]. ONNX checker passes, ORT passes.

5. **Explicit shape tensor via Python API** — `profile.set_shape_input("length_scale", ...)`
   marks it as shape tensor. Parse passes but `length_scale` still "unused at compile-time."

6. **builderOptimizationLevel=0** — same "unused" result.

7. **Final error with op=0**: `Could not find any implementation for node
   {ForeignNode[.../Cast_2...ONNXTRT_unsqueezeTensor_4667]}` — dynamic Unsqueeze
   not supported by TRT.

**Root cause**: The duration predictor computes L_mel (mel output frames) from
length_scale through a 20+ hop computation chain. TRT's static shape analysis
can't resolve this data-dependent computation at compile time. Even with all
surgery fixes, TRT can't connect `length_scale` to the mel output shape.

**Implemented solution**: Split the ONNX at the duration/decoder boundary.
Run the duration/encoder/noise front graph on ORT-CPU to produce `mu`, `mask`,
and `z0`, then run each ODE estimator step as a TensorRT BF16 engine over
dynamic mel length `T`. The split avoids the data-dependent shape chain while
moving the expensive estimator block to TensorRT.

## Quick Start (for the next person)

### Build TRT engines

```bash
# On Jetson Orin with TRT 10.3:
# 1. ONNX surgery (can be done on any machine)
python3 scripts/surgery_paraformer_decoder.py \
  --input decoder.onnx --output decoder-trt.onnx --validate

# 2. Build encoder
/usr/src/tensorrt/bin/trtexec --onnx=encoder.onnx \
  --saveEngine=paraformer_encoder_dp4_400.plan \
  --minShapes=speech:1x40x560,speech_lengths:1 \
  --optShapes=speech:1x80x560,speech_lengths:1 \
  --maxShapes=speech:1x400x560,speech_lengths:1 \
  --memPoolSize=workspace:2048 --skipInference

# 3. Build decoder (CRITICAL: use BF16, NOT FP16)
/usr/src/tensorrt/bin/trtexec --onnx=decoder-trt.onnx \
  --saveEngine=paraformer_decoder_bf16.plan --bf16 \
  --minShapes=enc:1x1x512,acoustic_embeds:1x1x512,pad_mask:1x1,enc_pad_mask:1x1 \
  --optShapes=enc:1x40x512,acoustic_embeds:1x10x512,pad_mask:1x10,enc_pad_mask:1x40 \
  --maxShapes=enc:1x400x512,acoustic_embeds:1x40x512,pad_mask:1x40,enc_pad_mask:1x400 \
  --memPoolSize=workspace:1024

# 4. Split Matcha and build estimator engines
python3 scripts/split_matcha_trt.py \
  --input /opt/models/matcha-icefall-zh-en/model-steps-3.onnx \
  --output-dir /opt/models/matcha-icefall-zh-en/onnx \
  --verify

ONNX_DIR=/opt/models/matcha-icefall-zh-en/onnx \
OUT_DIR=/opt/models/matcha-icefall-zh-en/engines \
WS=1024 \
bash scripts/build_matcha_estimator_trt.sh
```

### Deploy

```bash
docker build -f deploy/docker/Dockerfile.jetson \
  --build-arg LANGUAGE_MODE=zh_en -t seeed-local-voice:jetson-zh_en .

docker run -d --name seeed-voice --network=host --runtime=nvidia \
  -e SEEED_LOCAL_VOICE_PROFILE=jetson-zh-en \
  -v seeed-models:/opt/models \
  seeed-local-voice:jetson-zh_en
```

### Verify

```bash
curl http://localhost:8621/health

```

## Known Issues

1. **Full Matcha acoustic TRT remains blocked** — use split TRT mode instead
2. **Short Chinese greetings are a weak ASR exact-match gate** — `你好` often
   returns `好嗯`/`你好嗯` even with full ORT acoustic; validate with longer
   semantic phrases such as `欢迎使用本地语音助手`
3. **Encoder TRT may NaN on edge-case audio** — falls back to ORT-CPU automatically
4. **Engine files in Docker volume** (`seeed-nano-models`) survive reboots; `/tmp` engines do not
