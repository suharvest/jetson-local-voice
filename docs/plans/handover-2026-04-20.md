## 悬案

### 悬案 1 — Prefill 44ms vs 17ms 是测量口径问题，不是 regression

- `ttft_meas.py` 的 Python early-break（读够 4KB 后 close 连接）会导致 C++ 端清理时下一次 prefill 看起来要 ~44ms
- 用 curl 完整消费 stream 实测 prefill 稳定 **15.8-18.1ms**（和昨早 17ms 基线一致）
- 不需要 sync-primitive fix（昨晚的 `858c11e` 已 revert）
- 新 `/tmp/test_v2v_real.py` 用 `iter_content(chunk_size=1024)` + 累积 ≥5KB 才计时，相对靠谱

### 悬案 2 — V2V 场景稳定 crash（最优先）

**触发序列**：
1. `/tts` offline × N → OK
2. WebSocket `/asr/stream` → audio in → finalize → transcript OK
3. `/tts/stream` → OK
4. 下次 `/tts` offline → **CRASH**

**崩在**：`tts_trt_engine.cpp:807`，`cudaStreamSynchronize(stream_)` 报 "operation not permitted when stream is capturing"（CUDA 错误号 cudaErrorIllegalState）。

**pre-existing**：revert sync fix 后同样崩，md5 `f6cb7c2d`（T6 那版）就已经有这个 bug。

**Codex 诊断 (task `ada173b5263be89a0`) 已出 top 3 假说，排序**：

#### H-A（最可能）— Talker `DecodeStep` capture state 跨请求存活

- Talker 唯一的 capture 窗口在 `tts_trt_engine.cpp:959`（BeginCapture）→ `:966`（EndCapture）
- 流式每帧都调 `talker_->DecodeStep`（`tts_pipeline.cpp:1202-1203`）
- Pybind 在 `tts_binding.cpp:291-294` 释放 GIL；客户端断开、生成器提前退出、exception 任一触发，C++ 可能正处 capture 中但 EndCapture 永不调 → stream 永久 capturing
- 下次 `Prefill` 调 `cudaStreamSynchronize(stream_)` 立刻崩
- **无 RAII 保护**：BeginCapture..EndCapture 之间没有 try/catch 或 destructor 兜底

**实验**：在 Prefill 入口加 `cudaStreamIsCapturing(stream_, &status)` 打日志，在 DecodeStep 入口也加。临时用 mutex 序列化 TTSPipeline 所有访问——如果崩消失就是 leftover-capture。

**修法**：RAII CaptureGuard（unwind / destructor 里调 `cudaStreamEndCapture` 丢弃 graph）+ `Reset()` 检查 `cudaStreamIsCapturing` 强制 end capture。

#### H-C（次可能）— ASR 解码器共用 `TRTTalkerEngine` 类

- ASR 在 `qwen3_asr.py:583-587` 创建 `TRTTalkerEngine` 并启用 CUDA Graph
- 同一 pybind 类（`tts_binding.cpp:55-57`）
- 每个实例有独立 `stream_`（`tts_trt_engine.cpp:98`），理论上不互相污染——但 **若 ASR finalize 路径在 capture 中抛异常**，ASR 自己的 stream 永久 capturing
- 如果 ASR 和 TTS 共用 **同一 instance**（而非独立实例），立即崩

**实验**：`qwen3_asr.py:586` 加 `self._decoder.enable_cuda_graph(False)` 关掉 ASR CUDA Graph 重跑 V2V，若崩消失→ASR 嫌疑确认

**修法**：同 H-A 的 RAII guard，并审计 ASR/TTS 实例独立性

#### H-B（次次可能）— CP graph cache 的 T1 D2D copy 路径

- CP capture 边界 `tts_trt_engine.cpp:1828-1843`，常量表 D2D copy 在 capture 内（:1831-1837）
- CP 有自己独立 stream，**不能直接导致 Talker 崩**——除非跨 stream 事件同步意外传播 capture 状态
- 风险但不解释当前症状

**实验**：`CP_GRAPH=0` 环境变量重跑，同时打两个 stream 的 capture status 日志

### 悬案 3 — ASR "流式"实际是 accumulate-then-finalize

Memory `project_streaming_asr_deploy.md` 里早就记录。日志 `enc=56ms dec=291ms`、finalize 时间随音频长度线性增长：

| 音频长 | ASR finalize |
|---|---|
| 1.0s | 111ms |
| 1.4s | 347ms (typical) |
| 4.4s | 752ms |

真流式应该近似常数。当前实现**累积到 EOS 才大批量解码**，用户说越长等越久。

不是本轮工作目标，但值得另起 branch 诊断 `backends/qwen3_asr.py` 的 `accept_waveform` + `finalize` 流程，看是真解码被推迟还是只是 `finalize` 里二次跑 encoder。
