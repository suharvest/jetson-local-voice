## Analysis

### Observed Patterns

1. **Severe truncation**: ASR returns only 2-4 characters regardless of input length
   - "你好" → "当前。" (2 chars vs 2 chars, but wrong)
   - "今天天气真不错，我们一起去公园散步吧。" → "是現" (2 chars vs 21 chars)

2. **Hallucination**: ASR outputs completely unrelated text
   - All outputs contain "当前", "是" which are not in any ground truth
   - This suggests the ASR is not actually processing the audio content

3. **Consistent wrong output**: Different inputs produce similar outputs
   - sample_0000, 0001, 0003 all produce "当前" variants
   - sample_0002, 0004 produce "是" variants

### Potential Root Causes

Based on the spec (`docs/specs/asr-streaming-refactor.md`), these symptoms match known bugs:

- **Bug A (首字丢失)**: Finalize returns `_archive_text + _prev_text` which may lose content
- **Bug C (重复幻觉)**: `_archive_text` accumulative output causing repetition
- **Bug D (英文挂)**: `ASR_TEXT` prompt missing for auto mode - may also affect zh
- **Bug E (早期 EOS)**: Audio context cleared prematurely

The **consistent wrong output** suggests the ASR may be:
1. Not receiving the audio properly via WebSocket
2. Processing cached/stale audio context
3. Using wrong encoder state (Bug B: stale speculative embedding)

## Verdict

**Wrapper has SYSTEMIC issues** ❌

**Median CER: 100.00%** (far exceeds the 10% threshold)

The ASR wrapper shows **severe accuracy problems**:
- All samples have >90% CER
- ASR outputs are unrelated to audio content
- Output is severely truncated (2-4 chars regardless of input length)

### Caveats

This evaluation used **TTS-generated samples**, not real human speech. However:
- TTS-generated audio is typically cleaner than human recordings
- If ASR cannot recognize clean synthetic audio, it likely cannot recognize real human speech
- The spec notes: "不要用 TTS→ASR round-trip（TTS 输出噪声）" - but results here show the ASR is failing completely, not just having artifacts

### Recommendation

**Immediate action required**: Investigate root cause via codex analysis.

The ASR is outputting completely wrong text ("当前", "是現") for all inputs, indicating:
1. WebSocket audio transmission may be corrupted
2. Encoder/decoder state management is broken
3. Prompt construction is incorrect

Before latency optimization, must fix:
1. Verify WebSocket receives audio correctly (add logging in `_process_chunk`)
2. Check `_build_prompt` always includes `ASR_TEXT` (Bug D)
3. Eliminate speculative encoding (Bug B)
4. Fix `_segments` state management in finalize (Bug A, C, E)

**Next steps**:
1. Run codex:rescue analysis on `app/backends/qwen3_asr.py` per spec
2. Implement PR1 fixes (indentation, speculative encoding, prompt)
3. Re-run evaluation with real human speech samples
